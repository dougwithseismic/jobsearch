## Detailed Case Study: Contra's Indy.ai Chrome Extension

### The Problem (Inventory Bottleneck)

Contra, a commission-free freelance marketplace, faced a chicken-and-egg liquidity problem. They needed massive volume of high-quality job opportunities to attract freelancers. Traditional scraping approaches failed because:
- LinkedIn and X block data center IPs
- Maintaining proxy rotation infrastructure is prohibitively expensive
- Generic scrapers lack social graph context

### Solution: Distributed Intelligence

Inverted the scraping model by leveraging users themselves as distributed edge nodes.

**System Architecture:**
1. **Passive Read** - Users browse naturally
2. **Content Script** - Analyzes DOM
3. **Background Worker** - Processes data
4. **LLM Analysis Pipeline** - Extracts opportunities
5. **Contra Database** - Stores results

### Engineering Spotlight: Invisible Authentication

Implemented seamless session detection:
- Monitors network requests to specific domains
- Detects active sessions passively
- Never stores or transmits session cookies

### Impact

- **5 Million+ Opportunities Found** - 10x inventory growth
- **30,000+ Active Nodes** - Created massive distributed residential proxy network
- **High Fidelity** - Real user feeds contain high-intent opportunities
- **4.6/5 Star Rating** - High retention through genuine utility

### Why This Matters

Paradigm shift from **Centralized Extraction** to **Edge Discovery**. By distributing workload to the edge (user's browser), they bypassed scaling limitations and built a self-reinforcing data moat.

## Detailed Case Study: Snacker.ai - AI Video Editing Platform

### The Problem (Browser Bottleneck)

Video editing traditionally required:
- Heavy desktop applications
- Expensive server-side rendering farms
- Complex engineering challenges (memory limits, codec support, JavaScript single-thread)

Goal: Zero UI editing - take raw webcam feed, analyze with LLM, produce polished 20-second video without a render farm.

### Solution: Hybrid Client-Edge Architecture

Split workload:
- Heavy lifting on client (user's GPU via WebCodecs)
- Intelligence on edge (transcription and edit decisions)

**System Architecture:**
1. **Dual Stream Approach**
   - High-bitrate stream for final output
   - Low-latency proxy stream for real-time transcription
2. **Webcam Feed** - User records
3. **Stream 1: High Res** - Sent to WebCodecs
4. **Stream 2: Proxy** - Sent to transcriber
5. **Edge Transcriber** - Converts speech to text
6. **LLM Analysis** - Creates Edit Decision List (EDL)
7. **Client Composer** - Applies cuts/zooms
8. **Final MP4** - Delivered to user

### Engineering Spotlight: Bypassing iOS Restrictions

Apple restricts auto-playing multiple video elements. Solution: HLS (HTTP Live Streaming) generator.
- Generate .m3u8 playlist manifest on-the-fly
- Browser player treats as single continuous stream
- Seamless cutting without re-encoding
- Zero penalty

### The LLM Director

Fine-tuned model understands pacing and emphasis. Outputs Edit Decision List (EDL) JSON with cuts, removes, zoom instructions.

### Business Impact

- **Cost Efficiency:** 90% processing on user's device
- **Speed:** 20-second turnaround (10x faster than competitors like Descript)
- **Scalability:** Stateless, infinitely horizontally scalable

### Results

- **5,000+ users** in 3 months
- **20 paying customers**
- **20-second video turnaround**
- **Built with:** TypeScript, React, Hono, Supabase, WebCodecs API
